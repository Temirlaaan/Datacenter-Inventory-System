# Bug report: `GET /api/v1/meta/statuses` → 500 Internal Server Error

**Дата обнаружения:** 2026-06-11, ~20:12 Астана (**15:12:08 GMT** — время для поиска в серверных логах)
**Источник:** мобильный клиент DC Inventory (Android), реальное устройство, VPN, валидный токен
**Severity:** high — блокирует на мобильном клиенте экраны **редактирования** и **создания** устройства (оба запрашивают справочник статусов)

## Симптом

Эндпоинт `GET /api/v1/meta/statuses` стабильно возвращает `500 Internal Server Error`
с телом `text/plain` (21 байт, строка `Internal Server Error`) вместо JSON.

## Трейс с устройства (okhttp logcat, последовательные запросы одной сессии)

| # | Запрос | Ответ |
|---|--------|-------|
| 1 | `GET /api/v1/devices/238` | **200**, корректный JSON `{data, version}` |
| 2 | `GET /api/v1/meta/device-form` | **200**, корректный JSON, `version: "2026-06-04.1"` |
| 3 | `GET /api/v1/meta/statuses` | **500**, `content-type: text/plain; charset=utf-8`, тело `Internal Server Error` |

Заголовки запроса №3:

```
GET https://qr-dc.t-cloud.kz/api/v1/meta/statuses
X-Request-ID: 84272616-7e1c-4c1a-b99f-64e599a6edc5
Authorization: Bearer <валидный токен — запросы №1–2 с ним же прошли>
```

Заголовки ответа №3 (важно — чего в нём НЕТ):

```
HTTP/1.1 500
server: nginx
date: Thu, 11 Jun 2026 15:12:08 GMT
content-type: text/plain; charset=utf-8
content-length: 21
strict-transport-security / x-frame-options / x-content-type-options / ... (security-заголовки на месте)
```

## Почему это необработанное исключение в хендлере, а не клиентская проблема

1. Запросы №1–2 в той же цепочке (тот же токен, та же сессия, тот же IP) вернули 200 —
   аутентификация, VPN и сеть в порядке.
2. Тело — голый `text/plain "Internal Server Error"`, а не структурированный JSON-конверт
   ошибок (`{error: {code, message, ...}}`), которым бэкенд отвечает на все обрабатываемые
   ошибки. Это **дефолтный ответ Starlette/FastAPI на unhandled exception**.
3. В ответе **отсутствует `x-request-id`**, который бэкенд эхом возвращает во всех остальных
   ответах (и 2xx, и 4xx) — request-id middleware не доработал до конца, цепочка прервана
   исключением.
4. Без авторизации эндпоинт возвращает корректный 401 → auth-middleware жив, падение
   происходит **внутри хендлера** после аутентификации.

## Сравнение с соседними эндпоинтами

- `GET /health` → 200
- `GET /api/v1/meta/sites` (без токена) → 401 (корректный)
- `GET /api/v1/meta/statuses` (без токена) → 401 (корректный) — падает только с токеном,
  т.е. на пути выборки данных
- `GET /api/v1/meta/device-form` (с токеном) → 200

## Гипотезы (куда смотреть)

1. **Поход в NetBox за choices статусов** — хендлер statuses, в отличие от sites/racks,
   вероятно дёргает у NetBox метаданные поля `status` (например, OPTIONS/choices на
   `/api/dcim/devices/`). Изменение версии NetBox или формата ответа → KeyError/ValidationError.
2. **Кэш-слой** (TTL 5 минут по архитектуре): протухший/битый формат закэшированного
   значения, падение при десериализации.
3. **Pydantic-валидация ответа** `MetaStatus {value, label}`: NetBox вернул статус
   с неожиданным полем/null.

Грепнуть серверные логи по времени `2026-06-11 15:12:08 GMT` и/или по
`X-Request-ID: 84272616-7e1c-4c1a-b99f-64e599a6edc5` (если входящий request-id логируется).

## Ожидаемое поведение (acceptance)

1. `GET /api/v1/meta/statuses` с валидным токеном → `200` и JSON-массив
   `[{"value": "active", "label": "Active"}, ...]`.
2. Любая внутренняя ошибка хендлера → стандартный JSON-конверт ошибки (не голый text/plain)
   с эхом `x-request-id` — мобильный клиент парсит конверт и показывает человеческое сообщение.

## Влияние на мобильный клиент

- Экран «Редактировать устройство»: загрузка падает на шаге справочников → пользователь видит ошибку.
- Экран «Создать устройство» (новый): тот же справочник → тоже заблокирован.
- Остальные флоу (скан, просмотр, привязка, комментарии, смены) не затронуты.
