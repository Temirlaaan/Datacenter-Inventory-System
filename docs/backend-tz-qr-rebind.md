# ТЗ: перепривязка QR-метки — `POST /api/v1/qr/{qr_id}/rebind`

**Для:** бэкенд qr-dc (FastAPI-прослойка над NetBox)
**Инициатор:** мобильный клиент DC Inventory
**Приоритет:** средний (реальный полевой кейс, но есть обходной путь — переклейка меток)

## Бизнес-кейс

На части оборудования (коммутаторы и пр.) физически некуда клеить QR — метку
клеят на раму стойки на уровне устройства. Когда такие устройства физически
меняют местами, метка остаётся на старом уровне и продолжает указывать на
устройство, которого там больше нет.

Сейчас выйти из состояния BOUND можно только через `retire` (терминальный) —
перепривязать метку на другое устройство нельзя ни одним эндпоинтом. Печатать
и клеить новую метку из-за каждого свапа — расточительно.

## Контракт

```http
POST /api/v1/qr/{qr_id}/rebind
Idempotency-Key: <uuid>            ← опционально, но мобильный клиент шлёт всегда

{
  "device_id": 4711,               // новое устройство
  "version": "2026-06-… (last_updated нового устройства)",
  "reason": "Свап коммутаторов sw-01 и sw-02 при замене"   // ОБЯЗАТЕЛЬНОЕ, 1..2000
}
```

Ответ `200` — тот же формат, что у `bind`/`lookup`: `{qr, device}`, где
`device` — новое привязанное устройство (с новым `version`).

## Семантика

1. QR должен быть в статусе **BOUND** (для FREE есть `bind`; RETIRED — терминален).
2. Новое устройство **не должно** иметь свой QR (`custom_fields.qr_id` пуст) —
   иначе 409, мы не позволяем молча «украсть» метку у другого устройства.
3. Атомарно (насколько позволяет NetBox, в порядке: очистить → назначить):
   - у старого устройства очищается `custom_fields.qr_id` (+journal-запись
     «QR {id} отвязан: {reason}»);
   - у нового устройства проставляется `custom_fields.qr_id` с optimistic
     concurrency по присланному `version` (+journal-запись);
   - регистр QR обновляет `bound_to_device_id`, `bound_at`.
   - Если второй шаг упал после первого — компенсация (вернуть qr_id старому)
     либо консистентное состояние «QR временно ничей» с пометкой в аудите —
     на усмотрение исполнителя, но БЕЗ молчаливой потери привязки.
4. **Аудит обязателен**: отдельный action `qr.rebind` с qr_id, старым device_id,
   новым device_id, reason, пользователем, сменой.
5. Требуется активная смена (`409 NO_ACTIVE_SHIFT` — как у всех записей).
6. Idempotency-Key — как у `bind` (повтор с тем же ключом → исходный ответ).

## Права

Решение за владельцем процесса, рекомендация: разрешить роли `dcinv-engineer`
(не только admin) — кейс полевой, гонять админа к стойке нерационально.
Компенсация риска: обязательный `reason` + полный аудит.

## Ошибки

| Код | HTTP | Когда |
|---|---|---|
| `QR_NOT_FOUND` | 404 | нет такого qr_id |
| `QR_NOT_BOUND` | 409 | QR в статусе FREE (нужен bind) или RETIRED |
| `DEVICE_NOT_FOUND` | 404 | нет нового устройства |
| `DEVICE_ALREADY_BOUND` | 409 | у нового устройства уже есть QR (вернуть его qr_id в detail) |
| `SAME_DEVICE` | 409 или 200-noop | device_id совпадает с текущим — на усмотрение |
| `VERSION_CONFLICT` | 409 | новое устройство изменилось; клиент перечитает и переспросит |
| `NO_ACTIVE_SHIFT` | 409 | смена не открыта |
| (валидация) | 422 | reason пустой/длиннее 2000 |

## Мобильный клиент (после готовности)

В меню профиля привязанного устройства появится «Перепривязать метку»:
поиск нового устройства (как в bind) → обязательное поле «Причина» →
подтверждение с именами старого и нового устройств → rebind → обновлённый
экран. Реализация на мобильной стороне — ~полдня, начнём по факту эндпоинта.
